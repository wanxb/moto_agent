# 成功指标

> 衡量产品是否成功、功能是否值得做。MVP 阶段为单用户，指标以"自我验证价值"为主；Phase 3 多用户后转向规模化指标。

---

## 北极星指标（North Star）

> **每周成功记录的加油次数（Weekly Logged Fill-ups）**

理由：它同时反映「用户在用」+「记录足够顺畅以致愿意持续用」。一次成功记录 = 用户感知到价值的最小闭环。

---

## 指标体系（AARRR 裁剪版）

MVP 单用户阶段只关注**激活**与**留存/参与**，获客/变现留待 Phase 3。

### 激活（Activation）

| 指标 | 定义 | MVP 目标 |
|------|------|---------|
| 首次成功记录时长 | 从 `/start` 到第一条成功 `log_fuel` 的时间 | < 2 分钟 |
| 首次记录成功率 | 首次尝试记录即成功（无需多轮澄清失败）占比 | > 80% |

### 参与/留存（Engagement / Retention）

| 指标 | 定义 | MVP 目标 |
|------|------|---------|
| 周记录次数 | 北极星 | ≥ 用户实际加油频次（不漏记） |
| 记录漏失率 | 实际加油却没记的比例（自评） | < 10% |
| 查询使用率 | 每周至少 1 次查询统计的周占比 | > 50% |

## 体验质量指标（产品健康度）

| 指标 | 定义 | 目标 | 数据来源 |
|------|------|------|---------|
| 解析准确率 | LLM 正确提取字段（升数/里程/价格）的比例 | > 95% | 抽样日志 + LLM 评测（见 [测试策略](../engineering/testing-strategy.md)） |
| 不必要澄清率 | 信息其实给全了却仍追问的比例 | < 5% | 抽样日志 |
| 端到端响应延迟 | 用户发消息到收到回复 | P50 < 3s, P95 < 5s | Workers 日志计时 |
| 错误记录率 | 记错数字/记错车导致需纠错的比例 | < 3% | 用户纠错次数统计 |

## 技术 SLO（服务等级目标）

对齐 [`../../PRD.md`](../../PRD.md) §10 非功能性需求：

| SLO | 目标 | 错误预算 |
|-----|------|---------|
| AI 可用性（含 fallback） | 99%+ | 主备双模型，DeepSeek 故障自动切 Claude |
| 服务可用性 | 99.9%+ | 依赖 Cloudflare Workers SLA |
| 月度成本 | < $1/月 | 超限即告警（DeepSeek 控制台限额） |

---

## 护栏指标（Guardrail — 不能为了增长牺牲）

| 护栏 | 红线 |
|------|------|
| 月度成本 | 不超过 $5/月（单用户阶段） |
| 数据完整性 | 零数据丢失；schema 只增不删 |
| 隐私 | 非白名单 `chat_id` 零数据访问 |
| 响应延迟 | P95 不超过 8s（超过即体验崩坏） |

## Phase 3+ 规模化指标（预留）

多用户开放后引入，现在不测：

- 周活跃用户（WAU）、次周留存、月留存曲线。
- 人均周记录次数、人均车辆数。
- 获客渠道转化、Dashboard 使用率。

---

## 如何采集（MVP 务实方案）

- **延迟/成功率/解析**：`console.log` 结构化埋点（`[metric] ...`）→ Cloudflare Logpush → 人工/脚本抽样。
- **漏记/纠错**：MVP 阶段靠用户（开发者本人）主观记录，不做自动埋点。
- **不过度工程化**：单用户阶段不引入 APM/分析平台，保持零成本。详见 [`../engineering/observability-ops.md`](../engineering/observability-ops.md)。
