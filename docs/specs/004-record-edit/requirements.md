# 需求：记录纠错 / 删除

> 规格 004 · 状态：✔️ 已实现（待线上部署） · 阶段：Phase 2 · 优先级：P2
> 关联：[roadmap](../../product/roadmap.md) · [design.md](design.md) · [tasks.md](tasks.md)

## 1. 问题陈述

用户记加油时会输错（里程、升数、价格、油品记错），目前无法修改或删除，错误数据会污染油耗统计和提醒判定（里程错得离谱会误触发提醒）。需要让用户用自然语言纠正最近的记录（[S11](../../product/personas.md#体验类)）。

## 2. 用户故事

- **US1（改字段）** 作为骑手，我想说"上一条里程改成 12680"，以便纠正最近一条加油记录的某个字段。
- **US2（删记录）** 作为骑手，我想说"删掉刚才那条加油记录"，以便移除错误记录。
- **US3（多车）** 作为多车骑手，纠错/删除作用于**指定/默认车**的最近一条加油记录。
- **US4（可恢复）** 作为用户，我希望删除是"软删除"，数据不真正丢失，以便万一误删可恢复（运维侧）。

## 3. 范围

**In Scope**
- 修改**最近一条加油记录**的字段：日期、里程、升数、总价、油品、备注（只改用户指明的字段）。
- 删除**最近一条加油记录**（软删除，读取时过滤，统计/提醒不再计入）。
- 复用 spec 001 车辆解析：作用于指定/默认车的最近记录。

**Out of Scope（本期不做）**
- 修改/删除**任意历史记录**（按 id/日期定位）——MVP 只动"最近一条"，指代最明确、最常用。
- 里程记录 / 维保记录 / 提醒的纠错——本期聚焦加油记录（油耗核心）。提醒已有 `cancel_reminder`。
- 撤销（undo）软删除的对话指令——恢复走运维（DB），不做对话入口。

## 4. 验收标准（Given / When / Then）

- **AC1（改单字段）** Given 最近一条加油里程 12580，When 用户说"上一条里程改成 12680"，Then 该记录 odometer 更新为 12680，其它字段不变。
- **AC2（改多字段）** When 用户说"上一条改成 9 升 88 块"，Then liters=9、price_total=88 同时更新。
- **AC3（无记录可改）** Given 无加油记录，When 用户要求修改，Then 提示"没有可修改的加油记录"。
- **AC4（未指明改什么）** When 用户只说"改一下上一条"没说改什么，Then 提示需要说明要改的字段。
- **AC5（软删除）** Given 最近一条记录，When 用户说"删掉最近一条"，Then 该记录被软删除，后续查询/统计/`get_last_record`/里程提醒判定都不再包含它。
- **AC6（删除后 last 回退）** Given 两条记录，When 删除最近一条，Then `get_last_record` 返回上一条。
- **AC7（按车）** Given 多车，When 用户说"通勤车上一条里程改成 X"，Then 只改"通勤车"的最近记录，不影响其它车。
- **AC8（无记录可删）** Given 无记录，When 要求删除，Then 提示"没有可删除的加油记录"。

## 5. 交互示例

```
用户：上一条里程改成 12680
Bot：✏️ 已修改最近一条加油记录（小绿）
     📍 里程：12,680 km
     ⛽ 10 L × ¥9.80/L = ¥98

用户：删掉刚才那条
Bot：🗑 已删除最近一条加油记录（小绿）
     2026-06-18 · 12,680 km · 10 L · ¥98
     （如需恢复请联系管理员）
```

## 6. 依赖与假设

- 依赖：spec 001（车辆解析）。
- 假设："最近一条"= 该车 `odometer` 最大且未删除的加油记录（与现有排序基准一致）。

## 7. 开放问题

| 问题 | 影响 | 待决 |
|------|------|------|
| 改里程后油耗变化 | 用户体验 | 修改后回显即时重算的本次油耗（与 log_fuel 一致）可选；MVP 先回显字段 |
| 软删占用 odometer 空间 | 提醒/统计正确性 | 所有读路径（含 getLatestOdometer）过滤 deleted_at，确保删后不影响 |
| 多条同时误删 | 数据安全 | 本期只删"最近一条"，单条操作，降低风险 |
