# 需求：图片上传与多模态分析

> 规格 020 · 状态：📝 草稿 · 阶段：Phase 4 · 优先级：P1
> 关联：[roadmap](../../product/roadmap.md) · [design.md](design.md) · [tasks.md](tasks.md)

## 1. 问题陈述

用户目前在加油站只能发文字或语音记录加油数据。遇到以下场景时，文字表达低效甚至不可能：

1. **仪表盘照片**——拍了里程数和故障灯，需要手动看+打字，还可能漏掉故障灯含义
2. **加油机照片/订单截图**——油号、单价、加油量、金额全在图片里，手打费时且容易输错
3. **摩托车照片**——想咨询保养/维修问题（"这个部位异响"、"链条是不是该换了"），文字描述不清楚
4. **故障灯亮起**——用户不知道灯是什么意思、是否危险，需要即时诊断

**一句话**：让用户拍张照发过来，系统能"看懂"图片内容并自动完成记录、查询、诊断。

## 2. 用户故事

- 作为**摩托车骑手**，当我**拍了仪表盘照片发过去**，我希望**系统自动记录里程，如果有故障灯亮则告知原因和建议**。
- 作为**摩托车骑手**，当我**拍了加油机或订单截图发过去**，我希望**系统自动提取油号、单价、加油量、金额并完成记录**。
- 作为**摩托车骑手**，当我**拍了摩托车某部位照片并问"这个怎么保养"**，我希望**系统能从知识库查询相关回答**。
- 作为**有多辆车的骑手**，当我**拍了其中一辆车的照片**，我希望**系统能认出是哪辆车（或主动询问），不要乱猜**。
- 作为**摩托车骑手**，当我**拍了与摩托车无关的照片**，我希望**系统礼貌说明能力范围**。

## 3. 范围

**In Scope（v1）**

- Telegram 单张图片消息接收（`message:photo`）
- PWA Web 端图片上传（`POST /chat/image`）
- 图片经 LLM 多模态能力直接分析（不额外调用 OCR/视觉模型）
- 自动路由到支持多模态的模型（Claude Sonnet）
- 仪表盘：提取里程 + 识别故障灯 → 查知识库
- 加油机/截图：提取油号、单价、加油量、金额 → 调用 `log_fuel` 记录
- 摩托车照片：结合用户文字描述 → 调 `search_knowledge` 查询
- 多车识别：LLM 尝试从图片特征判断车辆，不确定则询问用户
- 非摩托车照片：回复"我只会看摩托车相关的图片哦"
- 图片附带 caption 时，caption 作为用户文字一起传给 LLM
- `Message` 类型扩展支持 `ContentBlock[]`（OpenAI 多模态兼容格式）
- 两个 LLM provider（DeepSeek + Anthropic）的传输层适配图片
- 图片**不持久化**到数据库/R2，处理完即丢弃（base64 内存态）
- 会话历史存文本（无图片，节省 KV 空间）

**Out of Scope（v1 不做）**

- 多张图片同时发送（media group）/ 相册——v1 只处理单张
- 视频/动图消息
- 图片持久化存储与历史查看
- 独立 OCR 管道（全靠 LLM 视觉）
- 训练/微调自定义视觉模型
- 图片编辑/裁剪/压缩的前端处理
- 基于图片的车型自动建档（如"拍一张车就添加到车辆列表"）
- 对已有图片重新分析/追问

## 4. 验收标准

- **AC1** Given Telegram 用户发送一张仪表盘照片（有里程数字 + 故障灯亮），When 系统处理，Then 自动记录里程并告知故障灯含义。
- **AC2** Given Telegram 用户发送一张加油小票照片，When 系统处理，Then 调用 log_fuel 记录加油数据。
- **AC3** Given PWA 用户通过 Web 上传一张摩托车照片并附带文字"这个螺丝该拧多紧"，When 系统处理，Then 调用 search_knowledge 查询。
- **AC4** Given 用户有多辆车且发了一张照片，When LLM 无法确认是哪辆车，Then 追问用户而不是擅自猜测。
- **AC5** Given 用户发了一张宠物照片，When 系统处理，Then 回复"只能处理摩托车相关的图片"。
- **AC6** Given 现有测试套件，When 本 spec 变更完成，Then `npm run type-check && npm test` 全绿。
- **AC7** Given 图片消息无附带文字，When 系统处理，Then 不阻断流程（不走 `extractText` 的空文本兜底）。

## 5. 交互示例

**场景 1：仪表盘 + 故障灯**
```
用户：[图片：仪表盘显示里程 15234km + 发动机故障灯亮]
Agent → search_knowledge("NS125LA 发动机故障灯亮")
       → log_mileage(odometer=15234)
Bot：📸 已记录里程：15,234 km
⚠️ 发动机故障灯亮了，知识库查到以下可能原因：
1. 氧传感器故障——建议检查传感器连接
2. 点火系统问题——检查火花塞
3. 建议尽快到维修站用诊断仪读取故障码
```

**场景 2：加油机**
```
用户：[图片：加油机显示 95# 8.23L ¥74.07]
Bot：⛽ 已记录加油：
- 油号：95
- 加油量：8.23 L
- 总价：¥74.07
- 里程：还没说，告诉我当前里程吧？
```

**场景 3：摩托车局部 + 问题**
```
用户：[图片：链条和链轮] "链条该紧了吗"
Bot → search_knowledge("NS125LA 链条调整")
📖 根据《NS125LA 用户手册》：
链条松弛度应在 20-30mm（上下摆动），
如果超出范围需要调整，建议每 1000km 检查一次。
要不要帮你设个链条保养提醒？
```

**场景 4：非摩托车图片**
```
用户：[图片：一只猫]
Bot：🐱 这张照片看起来不是摩托车相关的。
我只能处理摩托车相关图片——仪表盘、加油记录、维修保养等。有其他摩托车问题随时问我！
```

## 6. 依赖与假设

- 依赖：Anthropic Claude 多模态能力（`claude-sonnet-4-6` 支持图片输入）
- 依赖：Telegram Bot API `getFile` 端点（已有——voice 模式已使用）
- 假设：DeepSeek V4 Flash **不**支持图片输入（fallback 到 Claude）
- 假设：单张图片 base64 编码后 <15MB（Workers 128MB 内存 + LLM API 限制内）
- 假设：Telegram `message:photo` 的 `PhotoSize[]` 中至少有一个可下载

## 7. 开放问题

| 问题 | 影响 | 待决 |
|------|------|------|
| DeepSeek V4 Flash 是否支持多模态？若不支持是否回退到 Claude？ | 高——决定是否引入 `VisionAwareLLM` 分发层 | 设计采用保守策略：Claude 做图像 LLM，DeepSeek 只做文本 |
| Telegram 图片最大尺寸/分辨率？需不需要压缩？ | 中——影响 LLM API 费用和处理速度 | 设计中选择中等分辨率 PhotoSize，避免传原图 |
| 图片 caption 能否作为燃料记录的唯一文本输入？如果图片无 caption 系统如何处理？ | 中——影响 Pipeline 空文本判断 | 设计使 Pipeline 在图片消息时不依赖 `extractText` |
| KV 会话不存图片，用户后续追问"刚才那个图片里的问题"时 LLM 能否凭之前的文字回复回忆？ | 低——LLM 在之前的回复中已描述了图片内容，上下文足够 |
